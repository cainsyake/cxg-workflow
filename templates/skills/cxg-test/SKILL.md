# CXG Test - 智能测试生成

按代码类型路由 Codex 子进程生成测试方案与测试代码，并由主 Codex 整合验证。

## 使用方法

```bash
$cxg-test <测试目标>
```

## 上下文

- 测试目标：$ARGUMENTS
- 智能路由：后端 → `{{ROLE_TESTER}}`，前端 → `{{ROLE_TESTER_FRONTEND}}`，全栈 → 并行
- 遵循项目现有测试框架与风格

## 你的角色

你是**测试工程师**，负责编排测试分析、生成、整合与验证流程。

---

## 子进程调用规范

```text
Bash({
  command: "{{WRAPPER_BIN}} {{LITE_MODE_FLAG}}--backend codex - \"{{WORKDIR}}\" <<'EOF'
ROLE_FILE: <角色提示词路径>
<TASK>
需求：为以下目标生成测试
目标代码：<代码或文件范围>
需求描述：<增强后的测试需求>
要求：
1. 使用项目现有测试框架
2. 覆盖正常路径、边界条件、异常处理
</TASK>
OUTPUT: Unified Diff Patch（仅测试文件） + 测试策略说明
EOF",
  run_in_background: true,
  timeout: 3600000,
  description: "测试生成"
})
```

**角色提示词**：

| 测试视角 | 角色文件 |
|----------|----------|
| 后端/服务 | `{{ROLE_TESTER}}` |
| 前端/UI | `{{ROLE_TESTER_FRONTEND}}` |

**智能路由**：

| 代码类型 | 调用策略 |
|----------|----------|
| 后端 | 串行调用后端测试角色 |
| 前端 | 串行调用前端测试角色 |
| 全栈 | 并行调用两者并合并结果 |

**并行等待**：并行任务使用后台结果轮询机制，必须等待所有结果返回后再进入整合阶段。

**重要**：
- 必须指定 `timeout: 600000`，否则默认 30 秒容易超时。
- 若 10 分钟后仍未完成，继续轮询后台结果，绝对不要 Kill 进程。
- ⛔ **Codex 结果必须等待**：仅当已启动 Codex 子进程任务时，本规则生效。单次查询超时后必须继续轮询，禁止在结果返回前跳过该子进程结果。

## 执行工作流

### 阶段 0：Prompt 增强（可选）

`[模式：准备]`

按 `$cxg-enhance` 增强测试目标，明确测试边界和验收标准。

### 阶段 1：测试分析

`[模式：研究]`

1. 调用 `{{MCP_SEARCH_TOOL}}` 检索目标实现
2. 识别现有测试文件与测试框架配置
3. 判断代码类型：后端 / 前端 / 全栈
4. 评估覆盖缺口与高风险路径

### 阶段 2：路由生成

`[模式：生成]`

按代码类型调用对应测试角色，要求输出：
- 测试策略（覆盖范围与重点场景）
- 测试补丁（仅测试文件）

### 阶段 3：主 Codex 整合

`[模式：计划]`

1. 合并多路测试建议
2. 统一命名、结构与断言风格
3. 保证与项目现有测试范式一致

### 阶段 4：验证执行

`[模式：执行]`

1. 落盘测试文件
2. 运行新增/相关测试
3. 失败时定位并修复测试或实现问题

```text
    /\      E2E (10%)
   /--\     Integration (20%)
  /----\    Unit (70%)
```

## 关键规则

1. **测试行为，不测实现细节** - 关注输入输出与可观测行为
2. **复用现有模式** - 与仓库现有测试风格保持一致
3. **测试独立性** - 用例间不共享隐式状态
