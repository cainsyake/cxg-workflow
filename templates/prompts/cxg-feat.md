# CXG Feat - 智能功能开发

自动识别输入类型，完成规划、讨论迭代与实施交付。

## 使用方法

```bash
/prompts:cxg-feat <功能描述>
```

---

## 核心协议

- 与工具/子进程交互用英语，与用户沟通用中文。
- 子进程仅提供分析/原型建议，文件系统写入由主 Codex 执行。
- 输入可直接是新需求、计划迭代指令或执行请求。

---

## 子进程调用规范

**工作目录**：
- `{{WORKDIR}}`：替换为目标工作目录绝对路径。
- 多工作区时先检索定位目标目录。

**调用语法**：

```text
# 新会话
Bash({
  command: "{{WRAPPER_BIN}} {{LITE_MODE_FLAG}}--backend codex - \"{{WORKDIR}}\" <<'EOF'
ROLE_FILE: <角色提示词路径>
<TASK>
需求：<增强后的需求>
上下文：<前序阶段检索到的项目上下文、计划内容>
</TASK>
OUTPUT: 期望输出格式
EOF",
  run_in_background: true,
  timeout: 3600000,
  description: "功能协作"
})

# 复用会话
Bash({
  command: "{{WRAPPER_BIN}} {{LITE_MODE_FLAG}}--backend codex resume <SESSION_ID> - \"{{WORKDIR}}\" <<'EOF'
ROLE_FILE: <角色提示词路径>
<TASK>
需求：<增强后的需求>
上下文：<前序阶段检索到的项目上下文、计划内容>
</TASK>
OUTPUT: 期望输出格式
EOF",
  run_in_background: true,
  timeout: 3600000,
  description: "功能协作（复用会话）"
})
```

**角色提示词**：

| 阶段 | 后端/通用角色 | 前端/UI 角色 |
|------|----------------|--------------|
| 分析 | `{{ROLE_ANALYZER}}` | `{{ROLE_ANALYZER_FRONTEND}}` |
| 规划 | `{{ROLE_ARCHITECT}}` | `{{ROLE_ARCHITECT_FRONTEND}}` |
| 实施原型 | `{{ROLE_ARCHITECT}}` | `{{ROLE_FRONTEND}}` |
| 审查 | `{{ROLE_REVIEWER}}` | `{{ROLE_REVIEWER_FRONTEND}}` |

**并行调用**：使用 `run_in_background: true` 启动，随后通过运行时后台结果查询机制持续轮询。**必须等所有子进程返回后才能进入下一阶段**。

**重要**：
- 必须指定 `timeout: 600000`，否则默认只有 30 秒会导致提前超时。
- 若 10 分钟后仍未完成，继续轮询后台结果，绝对不要 Kill 进程。
- ⛔ **Codex 结果必须等待**：仅当已启动 Codex 子进程任务时，本规则生效。Codex 执行时间较长（5-15 分钟）属于正常。单次查询超时后必须继续轮询，**绝对禁止在 Codex 未返回结果时直接跳过或继续下一阶段**。已启动的 Codex 任务若被跳过 = 浪费 token + 丢失结果。

## 核心工作流程

### 阶段 1：输入类型判断

**每次交互必须先声明**：`我判断此次操作类型为：[具体类型]`

| 类型 | 判断信号 | 动作 |
|------|----------|------|
| 需求规划 | 实现/开发/新增/设计 | 进入阶段 2 |
| 讨论迭代 | 调整/优化/计划文件路径 | 读取现有计划后进入阶段 2.3 |
| 执行实施 | 开始实施/执行计划 | 进入阶段 3 |

### 阶段 2：需求规划流程

#### 2.0 Prompt 增强

按 `/prompts:cxg-enhance` 的逻辑增强 $ARGUMENTS，后续阶段统一使用增强需求。

#### 2.1 上下文检索

调用 `{{MCP_SEARCH_TOOL}}` 检索相关代码、组件和依赖关系。

#### 2.2 任务类型判断

| 任务类型 | 判断依据 | 调用流程 |
|----------|----------|----------|
| **前端** | 页面、组件、UI、样式、布局 | ui-ux-designer → planner |
| **后端** | API、接口、数据库、逻辑、算法 | planner |
| **全栈** | 同时包含前后端 | ui-ux-designer → planner |

#### 2.3 子进程协作规划

- 前端/全栈：先调用 `{{ROLE_ANALYZER_FRONTEND}}`，产出 UI/UX 方案
- 所有任务：调用 `{{ROLE_ARCHITECT}}` 或 `{{ROLE_ARCHITECT_FRONTEND}}`，产出可执行计划

#### 2.4 保存计划

文件命名规则：
- 首次规划：`.codex/plan/<功能名>.md`
- 迭代版本：`.codex/plan/<功能名>-v2.md`、`.codex/plan/<功能名>-v3.md`

#### 2.5 交互确认

规划完成后提供选项：
- 开始实施
- 继续讨论调整
- 重新规划
- 仅保存计划

### 阶段 3：执行实施流程

1. 读取计划（优先用户指定路径）
2. 提取任务分类：前端 / 后端 / 全栈
3. 路由子进程原型：
   - 前端：`{{ROLE_FRONTEND}}`
   - 后端：`{{ROLE_ARCHITECT}}`
   - 全栈：并行调用两者
4. 主 Codex 重构并落盘，完成后执行必要验证
5. 询问是否运行 `/prompts:cxg-review` 做正式审查

---

## 关键规则

1. **强制声明类型** - 每次交互先声明操作类型
2. **文档一致性** - 计划文档与执行结果保持同步
3. **信任规则** - 后端以后端角色结果为主，前端以前端角色结果为主
4. **透明沟通** - 所有判断和动作都要显式告知用户
